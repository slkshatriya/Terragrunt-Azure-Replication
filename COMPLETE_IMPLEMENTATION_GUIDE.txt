1.1  WHAT ARE WE BUILDING? (BIG PICTURE)
────────────────────────────────────────────────────────────────────────────────────────

SCENARIO:
  You already have some Azure resources running in Dev (created manually via Azure 
  Portal or some other way). Now you want to:
  
  1. Bring those existing Dev resources under Terraform/Terragrunt control
  2. Create exact copies in QA and Prod automatically
  3. Set up a CI/CD pipeline so future changes go through GitHub → Azure automatically
  4. Use OIDC authentication (no passwords stored anywhere!)

WHAT WE HAD (EXISTING DEV RESOURCES):
  ┌─────────────────────────────────────────────────────────────────┐
  │  Resource Group: MyResources-Dev (eastus)                      │
  │  ├── Automation Account: AutomateProcess (SystemAssigned)      │
  │  │   └── Runbook: Get-ServicesScript (PowerShell)              │
  │  ├── Logic App: DevLA                                          │
  │  └── Managed Identity: Dev-MI (User-Assigned)                  │
  └─────────────────────────────────────────────────────────────────┘

WHAT WE CREATED (QA + PROD):
  ┌─────────────────────────────────────────────────────────────────┐
  │  Resource Group: MyResources-QA (eastus)                       │
  │  ├── Automation Account: QA-AutomateProcess (eastus2*)         │
  │  │   └── Runbook: QA-Get-ServicesScript (eastus2*)             │
  │  ├── Logic App: QA-LA                                          │
  │  └── Managed Identity: QA-MI                                   │
  └─────────────────────────────────────────────────────────────────┘
  ┌─────────────────────────────────────────────────────────────────┐
  │  Resource Group: MyResources-Prod (eastus)                     │
  │  ├── Automation Account: Prod-AutomateProcess (westus*)        │
  │  │   └── Runbook: Prod-Get-ServicesScript (westus*)            │
  │  ├── Logic App: Prod-LA                                        │
  │  └── Managed Identity: Prod-MI                                 │
  └─────────────────────────────────────────────────────────────────┘
  
  * IMPORTANT: Azure only allows ONE Automation Account per subscription per region.
    Since Dev uses "eastus", QA must use "eastus2" and Prod must use "westus"
    (or any other DIFFERENT allowed region).

HOW IT WORKS (SIMPLE EXPLANATION):
  
  Think of it like a cookie cutter:
  - "Modules" are the cookie cutter shapes (templates)
  - "Environments" are different batches of cookies (dev/qa/prod)
  - Each batch uses the SAME cutter but with different sprinkles (names, tags)
  - "Terragrunt" is the baker that manages everything efficiently
  - "GitHub Actions" is the automated oven — push code, resources appear!
  - "OIDC" is the security badge — GitHub proves who it is to Azure, no passwords needed

1.4  STEP 1: CREATE AN AZURE AD APP REGISTRATION FOR OIDC
────────────────────────────────────────────────────────────────────────────────────────

WHAT IS OIDC?
  Instead of storing an Azure password/secret in GitHub, OIDC lets GitHub say 
  "Hey Azure, I'm GitHub Actions running for repo X" and Azure says "I trust that 
  repo, here's a temporary token." No passwords anywhere!

RUN THESE COMMANDS:

  # Step 1a: Create an App Registration
  az ad app create --display-name "github-actions-terragrunt"
  
  # Note down the "appId" from the output. Example: 9263e020-1d2e-44d5-a482-9e3e88977864
  # This is your CLIENT_ID — you'll need it everywhere!

  # Step 1b: Create a Service Principal for the app
  az ad sp create --id <YOUR_APP_ID>
  
  # Step 1c: Give it "Contributor" role on your subscription
  az role assignment create \
    --assignee <YOUR_APP_ID> \
    --role "Contributor" \
    --scope "/subscriptions/<YOUR_SUBSCRIPTION_ID>"

VERIFY IT WORKED:
  az role assignment list --assignee <YOUR_APP_ID> --query "[].{role:roleDefinitionName, scope:scope}" -o table

  Expected output:
    Role         Scope
    -----------  ---------------------------------------------------
    Contributor  /subscriptions/9c92613c-xxxx-xxxx-xxxx-xxxxxxxxxxxx

  *** SAVE YOUR APP_ID (CLIENT_ID) — You'll need it in Steps 10, 11, and 14! ***

1.5  STEP 2: CREATE A STATE STORAGE ACCOUNT IN AZURE
────────────────────────────────────────────────────────────────────────────────────────

WHAT IS STATE STORAGE?
  Terraform keeps track of what it has created in a "state file." This file MUST be 
  stored somewhere safe and shared (not on your laptop). We use an Azure Storage Account.

RUN THESE COMMANDS:

  # Step 2a: Create a Resource Group for state storage
  az group create --name terraform-state-rg --location eastus

  # Step 2b: Create a Storage Account (name must be globally unique, lowercase, no dashes)
  az storage account create \
    --name tfstatereplication26 \
    --resource-group terraform-state-rg \
    --location eastus \
    --sku Standard_LRS

  # Step 2c: Create a Blob Container inside it
  az storage container create \
    --name tfstate \
    --account-name tfstatereplication26

  # Step 2d: Give the Service Principal access to this storage
  # (Get the storage account resource ID first)
  STORAGE_ID=$(az storage account show --name tfstatereplication26 --resource-group terraform-state-rg --query id -o tsv)
  
  az role assignment create \
    --assignee <YOUR_APP_ID> \
    --role "Storage Blob Data Contributor" \
    --scope $STORAGE_ID

  *** NOTE: Replace "tfstatereplication26" with YOUR unique name ***
  *** The name must be 3-24 characters, lowercase letters and numbers only ***

1.6  STEP 3: CREATE THE PROJECT FOLDER STRUCTURE
────────────────────────────────────────────────────────────────────────────────────────

RUN THESE COMMANDS (PowerShell):

  # Create the project root
  mkdir Terragrunt-Azure-Replication
  cd Terragrunt-Azure-Replication

  # Create modules folders
  mkdir -p modules/resource-group
  mkdir -p modules/managed-identity
  mkdir -p modules/automation-account
  mkdir -p modules/runbook
  mkdir -p modules/logic-app

  # Create environment folders (repeat for each resource in each env)
  foreach ($env in @("dev", "qa", "prod")) {
    mkdir -p "environments/$env/resource-group"
    mkdir -p "environments/$env/managed-identity"
    mkdir -p "environments/$env/automation-account"
    mkdir -p "environments/$env/runbook"
    mkdir -p "environments/$env/logic-app"
  }

  # Create GitHub Actions folder
  mkdir -p .github/workflows

  # Initialize Git
  git init

1.7  STEP 4: CREATE THE TERRAFORM MODULES (REUSABLE TEMPLATES)
────────────────────────────────────────────────────────────────────────────────────────

Each module has 3 files:
  - main.tf       → The actual Azure resource definition
  - variables.tf  → Input parameters (what the caller provides)
  - outputs.tf    → Values that other modules can reference


MODULE 1: RESOURCE GROUP
─────────────────────────

FILE: modules/resource-group/main.tf
┌──────────────────────────────────────────────────────────────────┐
│  terraform {                                                     │
│    required_providers {                                           │
│      azurerm = {                                                 │

│        version = "~> 3.0"                                        │
│      }                                                           │
│    }                                                             │
│  }                                                               │
│                                                                  │

│  }                                                               │

────────────────────────────────────────────────────────────────────────────────────────
1.10  STEP 7: CREATE THE RESOURCE CONFIGS (PER ENVIRONMENT)
────────────────────────────────────────────────────────────────────────────────────────

WHAT IS THIS?
  Each resource in each environment has its own terragrunt.hcl file that says:
  - WHICH module to use (source)
  - WHAT values to pass (inputs)
  - WHICH other resources it depends on (dependency)

PATTERN (Same for every resource):
  1. include "root"   → Links to the parent terragrunt.hcl (backend + provider)
  2. locals           → Reads env.hcl for shared variables
  3. terraform source → Points to the module folder
  4. dependency       → Declares what this resource needs (e.g., runbook needs automation account)
  5. inputs           → The actual values (names, locations, tags)

FILE: environments/dev/resource-group/terragrunt.hcl
┌──────────────────────────────────────────────────────────────────┐
│  include "root" {                                                │

*** CRITICAL FOR AUTOMATION ACCOUNT (Single Subscription): ***
  Azure limit: ONE Automation Account per subscription per region!
  
  Since Dev uses "eastus", you MUST use different regions:
  - QA automation-account:   location = "eastus2"      (NOT from dependency!)
  - QA runbook:              location = "eastus2"      (MUST match its automation account)
  - Prod automation-account: location = "westus"
  - Prod runbook:            location = "westus"
  
  All other resources (RG, MI, Logic App) can stay in "eastus".
  
  Allowed regions (depends on subscription type):
  Free Trial: eastus, eastus2, westus, northeurope, southeastasia, japanwest
  Pay-As-You-Go: More regions available

1.13  STEP 10: SET UP OIDC FEDERATED CREDENTIALS IN AZURE
────────────────────────────────────────────────────────────────────────────────────────

THIS IS THE MOST IMPORTANT STEP! Without this, GitHub cannot authenticate to Azure.

WHAT ARE FEDERATED CREDENTIALS?
  They're like a trust agreement: "Azure, please trust tokens coming from GitHub 
  Actions when they say they're running for my specific repository."

YOU NEED TWO CREDENTIALS:

  1. For pushes/manual triggers (workflow_dispatch from main branch):

     az ad app federated-credential create --id <YOUR_APP_ID> --parameters '{
       "name": "github-actions-main",
       "issuer": "https://token.actions.githubusercontent.com",
       "subject": "repo:YOUR-USERNAME/Terragrunt-Azure-Replication:ref:refs/heads/main",
       "audiences": ["api://AzureADTokenExchange"]
     }'

  2. For pull requests:

     az ad app federated-credential create --id <YOUR_APP_ID> --parameters '{
       "name": "github-actions-pr",
       "issuer": "https://token.actions.githubusercontent.com",
       "subject": "repo:YOUR-USERNAME/Terragrunt-Azure-Replication:pull_request",
       "audiences": ["api://AzureADTokenExchange"]
     }'

  Replace YOUR-USERNAME and YOUR_APP_ID!

VERIFY:
  az ad app federated-credential list --id <YOUR_APP_ID> -o table

  You should see 2 entries.

*** COMMON MISTAKE: ***
  The "subject" must EXACTLY match what GitHub sends:
  - For branches:      repo:OWNER/REPO:ref:refs/heads/BRANCH
  - For pull requests: repo:OWNER/REPO:pull_request
  - For environments:  repo:OWNER/REPO:environment:ENV_NAME
  
  If using "environment:" in your workflow, use the environment subject.
  If NOT using "environment:" (our approach), use the branch/PR subjects.

1.15  STEP 12: IMPORT EXISTING DEV RESOURCES INTO TERRAFORM STATE
────────────────────────────────────────────────────────────────────────────────────────

WHAT IS IMPORTING?
  Your Dev resources already exist in Azure. Terraform doesn't know about them yet.
  "Import" tells Terraform: "Hey, THIS Azure resource is managed by THIS config."
  After importing, Terraform won't try to create them again — it'll just manage them.

BEFORE YOU IMPORT:
  You must be logged into Azure CLI:
    az login
    az account set --subscription <YOUR_SUBSCRIPTION_ID>

  You must set environment variables for OIDC (for local runs):
    # PowerShell:
    $env:ARM_CLIENT_ID = "<YOUR_APP_ID>"
    $env:ARM_SUBSCRIPTION_ID = "<YOUR_SUBSCRIPTION_ID>"
    $env:ARM_TENANT_ID = "<YOUR_TENANT_ID>"
    $env:ARM_USE_OIDC = "true"
    
    # OR for local dev, you can skip OIDC and just use Azure CLI auth by 
    # temporarily changing provider to NOT use OIDC. But for CI/CD it must use OIDC.

IMPORT COMMANDS (run from each resource's directory):

  1. RESOURCE GROUP:
     cd environments/dev/resource-group
     terragrunt import azurerm_resource_group.this /subscriptions/<SUB_ID>/resourceGroups/MyResources-Dev

  2. MANAGED IDENTITY:
     cd environments/dev/managed-identity
     terragrunt import azurerm_user_assigned_identity.this /subscriptions/<SUB_ID>/resourceGroups/MyResources-Dev/providers/Microsoft.ManagedIdentity/userAssignedIdentities/Dev-MI

     *** NOTE: "resourceGroups" must have capital G! Azure CLI sometimes shows lowercase. ***

  3. AUTOMATION ACCOUNT:
     cd environments/dev/automation-account
     terragrunt import azurerm_automation_account.this /subscriptions/<SUB_ID>/resourceGroups/MyResources-Dev/providers/Microsoft.Automation/automationAccounts/AutomateProcess

  4. RUNBOOK:
     cd environments/dev/runbook
     terragrunt import azurerm_automation_runbook.this /subscriptions/<SUB_ID>/resourceGroups/MyResources-Dev/providers/Microsoft.Automation/automationAccounts/AutomateProcess/runbooks/Get-ServicesScript

  5. LOGIC APP:
     cd environments/dev/logic-app
     terragrunt import azurerm_logic_app_workflow.this /subscriptions/<SUB_ID>/resourceGroups/MyResources-Dev/providers/Microsoft.Logic/workflows/DevLA

VERIFY THE IMPORT (CRITICAL STEP!):
  Go back to the environment root and run plan:
  
  cd environments/dev
  terragrunt run-all plan --terragrunt-non-interactive

  *** THE OUTPUT MUST SAY: "No changes. Your infrastructure matches the configuration." ***
  
  If it shows changes, your config doesn't match reality. Common fixes:
  - Tags are wrong (existing has none → use tags = {})
  - Description is wrong (existing has null → use description = null, not "")
  - Runbook content doesn't match (whitespace matters!)
  - Location casing (use lowercase: "eastus" not "EastUS")

1.17  COMMON ERRORS AND HOW TO FIX THEM
────────────────────────────────────────────────────────────────────────────────────────

ERROR 1: "a Tenant ID must be configured when authenticating with OIDC"
  CAUSE: GitHub secrets are not reaching the workflow.
  FIX:
    - Check secrets exist: gh secret list --repo YOUR-USERNAME/YOUR-REPO
    - Check you DON'T have "environment:" in the job definition
    - If using "environment:", secrets must be in GitHub Environment settings, 
      NOT repository settings

ERROR 2: "Only one account is allowed for your subscription per Region"
  CAUSE: Azure Automation Account limit — 1 per subscription per region
  FIX: Use different regions per environment:
    Dev = eastus, QA = eastus2, Prod = westus

ERROR 3: "Free Trial subscriptions cannot create accounts in this location"
  CAUSE: Your subscription type limits which Azure regions you can use
  FIX: Use only allowed regions. For Free Trial:
    eastus, eastus2, westus, northeurope, southeastasia, japanwest

ERROR 4: "OIDC: No matching federated credential found"
  CAUSE: The federated credential subject doesn't match what GitHub sends
  FIX: 
    - If NO "environment:" in workflow → use subject "repo:OWNER/REPO:ref:refs/heads/main"
    - If "environment: dev" in workflow → use subject "repo:OWNER/REPO:environment:dev"
    - They CANNOT be mixed!

ERROR 5: Import says "resource already exists in state"
  CAUSE: You already imported this resource
  FIX: Skip it, or remove from state first:
    terragrunt state rm azurerm_resource_group.this

ERROR 6: Plan shows changes after import (description, tags, content mismatch)
  CAUSE: Your config doesn't exactly match what's in Azure
  FIX:
    - Check the actual resource in Azure Portal or via CLI
    - Match EXACTLY: null vs "" matters, whitespace in runbook content matters
    - Use: az automation runbook show ... to check actual values

ERROR 7: "Error acquiring the state lock"
  CAUSE: A previous run crashed and left the state locked
  FIX: terragrunt force-unlock <LOCK_ID>

ERROR 8: "Backend configuration changed"
  CAUSE: You changed backend settings after init
  FIX: terragrunt init -reconfigure

2.1  WHAT CHANGES WITH MULTIPLE SUBSCRIPTIONS?
────────────────────────────────────────────────────────────────────────────────────────

In the real world, most organizations use SEPARATE Azure subscriptions per environment:

  ┌────────────────────────────────────────────────────────────────────┐
  │  SINGLE SUBSCRIPTION (Part 1)        MULTIPLE SUBSCRIPTIONS       │
  │  ─────────────────────               ──────────────────────       │
  │                                                                    │
  │  Subscription: SUB-123               Dev Sub:  SUB-DEV-111       │
  │  ├── MyResources-Dev                 ├── MyResources-Dev          │
  │  ├── MyResources-QA                                               │
  │  └── MyResources-Prod               QA Sub:   SUB-QA-222        │
  │                                      ├── MyResources-QA           │
  │  (All in one sub,                                                 │
  │   separated by RG name)              Prod Sub: SUB-PROD-333     │
  │                                      ├── MyResources-Prod         │
  │                                                                    │
  │                                      (Each env in its own sub)    │
  └────────────────────────────────────────────────────────────────────┘

KEY DIFFERENCES:
  ┌──────────────────────┬───────────────────────┬───────────────────────┐
  │  What Changes        │  Single Sub           │  Multiple Subs        │
  ├──────────────────────┼───────────────────────┼───────────────────────┤
  │  Subscription IDs    │  1 (same everywhere)  │  3 (one per env)      │
  │  GitHub Secrets      │  3 repo-level         │  3 per env OR dynamic │
  │  Provider config     │  No sub ID needed     │  Needs subscription_id│
  │  State storage       │  1 storage account    │  1 or 3 (your choice) │
  │  Service Principal   │  1 (Contributor on 1) │  1 or 3 (roles on 3)  │
  │  OIDC credentials    │  2 (main + PR)        │  2 (same if 1 SP)     │
  │  Automation Account  │  Different regions!   │  Same region OK!      │
  │  Folder structure    │  Same                 │  Same                 │
  │  Modules             │  Same                 │  Same                 │
  └──────────────────────┴───────────────────────┴───────────────────────┘

  BIG ADVANTAGE: With separate subscriptions, you CAN use the same region 
  ("eastus") for Automation Accounts in all environments! The 1-per-sub-per-region 
  limit applies per SUBSCRIPTION, not globally.

2.3  AZURE SETUP DIFFERENCES
────────────────────────────────────────────────────────────────────────────────────────

OPTION A: ONE SERVICE PRINCIPAL WITH ACCESS TO ALL SUBSCRIPTIONS (SIMPLEST)
  
  This is the easiest approach. One app registration, one SP, roles on all 3 subs.
  
  # Step 1: Create the app (same as Part 1)
  az ad app create --display-name "github-actions-terragrunt-multisub"
  az ad sp create --id <APP_ID>

  az role assignment create --assignee <APP_ID> --role "Contributor" --scope "/subscriptions/<DEV_SUB_ID>"
  az role assignment create --assignee <APP_ID> --role "Contributor" --scope "/subscriptions/<QA_SUB_ID>"
  az role assignment create --assignee <APP_ID> --role "Contributor" --scope "/subscriptions/<PROD_SUB_ID>"

  # Step 3: State storage — create in ONE subscription (e.g., Dev) or a shared "management" sub
  # Same as Part 1, but give access from all subs
  az role assignment create --assignee <APP_ID> --role "Storage Blob Data Contributor" --scope <STORAGE_ACCOUNT_ID>

OPTION B: SEPARATE SERVICE PRINCIPAL PER SUBSCRIPTION (MORE SECURE)
  
  Each environment gets its own SP with access to ONLY its subscription.
  More secure but more complex to manage.
  
  # Create 3 apps:
  az ad app create --display-name "github-terragrunt-dev"
  az ad app create --display-name "github-terragrunt-qa"

  az ad sp create --id <DEV_APP_ID>
  az ad sp create --id <QA_APP_ID>
  az ad sp create --id <PROD_APP_ID>
  
  # Assign roles (each to its OWN subscription only):
  az role assignment create --assignee <DEV_APP_ID> --role "Contributor" --scope "/subscriptions/<DEV_SUB_ID>"
  az role assignment create --assignee <QA_APP_ID> --role "Contributor" --scope "/subscriptions/<QA_SUB_ID>"
  az role assignment create --assignee <PROD_APP_ID> --role "Contributor" --scope "/subscriptions/<PROD_SUB_ID>"
  
  # All 3 SPs need access to state storage:
  az role assignment create --assignee <DEV_APP_ID> --role "Storage Blob Data Contributor" --scope <STORAGE_ID>
  az role assignment create --assignee <QA_APP_ID> --role "Storage Blob Data Contributor" --scope <STORAGE_ID>
  az role assignment create --assignee <PROD_APP_ID> --role "Storage Blob Data Contributor" --scope <STORAGE_ID>

2.10  COMPLETE MULTI-SUBSCRIPTION EXAMPLE (OPTION A — RECOMMENDED)
────────────────────────────────────────────────────────────────────────────────────────

Let's walk through the COMPLETE setup for Option A (one SP, multi-sub).

GIVEN:
  Dev Subscription:  aaaaaaaa-1111-1111-1111-aaaaaaaaaaaa
  QA Subscription:   bbbbbbbb-2222-2222-2222-bbbbbbbbbbbb
  Prod Subscription: cccccccc-3333-3333-3333-cccccccccccc
  Tenant ID:         dddddddd-4444-4444-4444-dddddddddddd
  App/Client ID:     eeeeeeee-5555-5555-5555-eeeeeeeeeeee
  State Storage in Dev subscription

STEP-BY-STEP:

  1. CREATE APP + SP + ROLES (run once):
     az ad app create --display-name "github-actions-terragrunt-multisub"
     # Note the appId → eeeeeeee-5555-5555-5555-eeeeeeeeeeee
     
     az ad sp create --id eeeeeeee-5555-5555-5555-eeeeeeeeeeee
     
     # Give Contributor on ALL 3 subs:
     az role assignment create --assignee eeeeeeee-5555-5555-5555-eeeeeeeeeeee --role "Contributor" --scope "/subscriptions/aaaaaaaa-1111-1111-1111-aaaaaaaaaaaa"
     az role assignment create --assignee eeeeeeee-5555-5555-5555-eeeeeeeeeeee --role "Contributor" --scope "/subscriptions/bbbbbbbb-2222-2222-2222-bbbbbbbbbbbb"
     az role assignment create --assignee eeeeeeee-5555-5555-5555-eeeeeeeeeeee --role "Contributor" --scope "/subscriptions/cccccccc-3333-3333-3333-cccccccccccc"

  2. CREATE STATE STORAGE (in Dev sub):
     az account set --subscription aaaaaaaa-1111-1111-1111-aaaaaaaaaaaa
     az group create --name terraform-state-rg --location eastus
     az storage account create --name tfstatemultisub01 --resource-group terraform-state-rg --location eastus --sku Standard_LRS
     az storage container create --name tfstate --account-name tfstatemultisub01
     
     # Give SP access to state:
     STORAGE_ID=$(az storage account show --name tfstatemultisub01 --resource-group terraform-state-rg --query id -o tsv)
     az role assignment create --assignee eeeeeeee-5555-5555-5555-eeeeeeeeeeee --role "Storage Blob Data Contributor" --scope $STORAGE_ID

  3. FILES — env.hcl (each includes subscription_id):
     
     # environments/dev/env.hcl
     locals {
       environment     = "dev"
       subscription_id = "aaaaaaaa-1111-1111-1111-aaaaaaaaaaaa"
       location        = "eastus"
       tags            = {}
     }
     
     # environments/qa/env.hcl
     locals {
       environment     = "qa"
       subscription_id = "bbbbbbbb-2222-2222-2222-bbbbbbbbbbbb"
       location        = "eastus"     # CAN be same region now!
       tags = { environment = "qa", managed_by = "terragrunt" }
     }
     
     # environments/prod/env.hcl
     locals {
       environment     = "prod"
       subscription_id = "cccccccc-3333-3333-3333-cccccccccccc"
       location        = "eastus"     # CAN be same region now!
       tags = { environment = "prod", managed_by = "terragrunt", criticality = "high" }
     }

  4. ROOT TERRAGRUNT.HCL (example for dev — change environment for qa/prod):
     
     locals {
       environment = "dev"
       env_vars = read_terragrunt_config("env.hcl")
     }
     
     remote_state {
       backend = "azurerm"
       config = {
         resource_group_name  = "terraform-state-rg"
         storage_account_name = "tfstatemultisub01"
         container_name       = "tfstate"
         key                  = "${local.environment}/${path_relative_to_include()}/terraform.tfstate"
         subscription_id      = "aaaaaaaa-1111-1111-1111-aaaaaaaaaaaa"   # <-- State sub (always Dev)
         use_oidc             = true
         client_id            = get_env("ARM_CLIENT_ID", "")
         tenant_id            = get_env("ARM_TENANT_ID", "")
       }
       generate = {
         path      = "backend.tf"
         if_exists = "overwrite_terragrunt"
       }
     }
     
     generate "provider" {
       path      = "provider.tf"
       if_exists = "overwrite_terragrunt"
       contents  = <<EOF
     provider "azurerm" {
       features {}
       use_oidc        = true
       subscription_id = "${local.env_vars.locals.subscription_id}"     # <-- Deploy sub (from env.hcl)
     }
     EOF
     }

     *** NOTE THE TWO DIFFERENT SUBSCRIPTION IDs: ***
     - remote_state.config.subscription_id = Where the STATE file lives (always the same)
     - provider.subscription_id = Where RESOURCES get created (different per env!)

  5. RESOURCE CONFIGS:
     *** EXACTLY THE SAME AS PART 1! ***
     The subscription routing happens in the provider, not in the resource configs.
     
     The Automation Account can now use the SAME region everywhere:
       location = dependency.resource_group.outputs.resource_group_location
     
     Because each sub has its own quota for Automation Accounts!

     AZURE_TENANT_ID        = dddddddd-4444-4444-4444-dddddddddddd

  7. OIDC FEDERATED CREDENTIALS (2):
     Same as Part 1 — branch + PR subjects.

  8. PIPELINES:
     *** EXACTLY THE SAME AS PART 1! ***

  9. DEPLOY:

2.11  STATE STORAGE OPTIONS FOR MULTI-SUBSCRIPTION
────────────────────────────────────────────────────────────────────────────────────────

OPTION 1: ONE CENTRAL STATE STORAGE (RECOMMENDED)
  
  Put the state storage in one subscription (e.g., Dev or a dedicated "management" sub).
  All environments write state to the same storage account, but different keys:
    dev/resource-group/terraform.tfstate
    qa/resource-group/terraform.tfstate
    prod/resource-group/terraform.tfstate
  
  Pros: Simple, one place to back up, one place to manage access
  Cons: If that sub is down, all envs are affected

OPTION 2: SEPARATE STATE STORAGE PER SUBSCRIPTION
  
  Each subscription has its own storage account:
    Dev sub  → tfstatedev01
    QA sub   → tfstateqa01
    Prod sub → tfstateprod01
  
  Each environment's root terragrunt.hcl points to its own storage:
    # dev/terragrunt.hcl
    storage_account_name = "tfstatedev01"
    subscription_id      = "aaaaaaaa-1111-1111-1111-aaaaaaaaaaaa"
    
    # qa/terragrunt.hcl
    storage_account_name = "tfstateqa01"
    subscription_id      = "bbbbbbbb-2222-2222-2222-bbbbbbbbbbbb"
  
  Pros: Complete isolation, more secure
  Cons: More storage accounts to manage

2.12  COMMON ERRORS SPECIFIC TO MULTI-SUBSCRIPTION
────────────────────────────────────────────────────────────────────────────────────────

ERROR 1: "The subscription '<ID>' could not be found"
  CAUSE: The service principal doesn't have access to that subscription.
  FIX: Add the Contributor role on the target subscription:
    az role assignment create --assignee <APP_ID> --role "Contributor" --scope "/subscriptions/<TARGET_SUB_ID>"

ERROR 2: Resources created in wrong subscription
  CAUSE: subscription_id not set in the provider block.
  FIX: Add subscription_id to the generate "provider" block:
    provider "azurerm" {
      features {}
      subscription_id = "${local.env_vars.locals.subscription_id}"
    }

ERROR 3: State backend can't access storage in a different subscription
  CAUSE: The remote_state subscription_id doesn't match where the storage account lives.
  FIX: In remote_state config, set subscription_id to the sub where your storage account is:
    subscription_id = "STORAGE-ACCOUNT-SUB-ID"  # NOT the deploy sub!

ERROR 4: "AuthorizationFailed" when accessing state
  CAUSE: SP doesn't have Storage Blob Data Contributor on the state storage.
  FIX: Assign the role:
    az role assignment create --assignee <APP_ID> --role "Storage Blob Data Contributor" --scope <STORAGE_ACCOUNT_RESOURCE_ID>

ERROR 5: Mixed up subscription IDs in remote_state vs provider
  CAUSE: Using the deployment sub ID in remote_state instead of the state-storage sub ID.
  REMEMBER: 
    remote_state.config.subscription_id = WHERE STATE LIVES
    provider.subscription_id            = WHERE RESOURCES GET CREATED
    These are DIFFERENT in multi-sub!

                             GLOSSARY
════════════════════════════════════════════════════════════════════════════════════════

  Terraform     = Tool that creates Azure resources from code files
  Terragrunt    = Wrapper around Terraform that reduces code duplication
  Module        = Reusable template for an Azure resource
  State File    = JSON file tracking what Terraform has created
  Backend       = Where the state file is stored (Azure Storage Account)
  Provider      = Plugin that talks to Azure (azurerm)
  OIDC          = Authentication method where GitHub proves its identity to Azure
  Service Principal = An identity for your app in Azure AD
  Federated Credential = Trust link between GitHub Actions and Azure AD
  Import        = Telling Terraform about an existing Azure resource
  Plan          = Preview of what Terraform would do (safe, read-only)
  Apply         = Actually execute the changes
  Destroy       = Delete all managed resources
  HCL           = HashiCorp Configuration Language (the .tf and .hcl file format)
  Dependency    = One resource needs another to exist first (e.g., Runbook needs Automation Account)
  Mock Outputs  = Fake values used during "plan" when the dependency doesn't exist yet
